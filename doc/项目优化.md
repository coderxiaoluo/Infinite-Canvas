# 项目优化指南（公司内部 / 小团队使用）

本文档面向**公司内部或小团队内网部署**场景，结合当前代码结构（`main.py` 约 1.48 万行、`canvas.js` 约 1.4 万行、JSON 落盘、内存任务队列）给出可落地的优化建议。

> **定位说明**：Infinite Canvas 是「可插拔 AI 引擎的工作流画布」，不是 Liblib.tv 那样的云端 SaaS。  
> 内部使用重点：**稳定、安全、可协作、易维护**，不必追求积分、公开分享、模板市场等平台化能力。

> **部署前提（本文默认）**：**无 GPU、无本地 ComfyUI**，算力全部走云端 API（APIMart、ModelScope、火山引擎、RunningHub 等）。部署步骤见 [`内部部署.md`](./内部部署.md)。

---

## 一、已有能力（勿重复建设）

对标 [Liblib.tv](https://www.liblib.tv/) 等产品时，以下能力**项目已具备**，不属于「缺失项」：

| 能力 | 说明 |
|------|------|
| **节点内嵌参数面板** | 视频 / API 生图 / MS 生成 / ComfyUI 等节点均在画布上直接配置参数并运行 |
| **视频节点完整控件** | Provider、模型、时长、比例、分辨率；优化提示词、1080p、水印、固定镜头、生成音频、全能参考、首尾帧等（见 `renderVideoBody()`） |
| **节点连线工作流** | Prompt + Image → 视频生成 → Output 等链路可直接编排 |
| **多 Provider 接入** | APIMart、OpenAI、Gemini、ModelScope、火山引擎、RunningHub 等（无 GPU 时以云端 API 为主） |
| **素材库 / Prompt 库** | 已有 asset-library、prompt-libraries |
| **多窗口协同基础** | WebSocket 广播 `canvas_updated`，保存冲突检测（409） |

**更正**：此前若将「节点内嵌参数」列为相对 Liblib 的缺口，属于**表述错误**。  
内部优化应聚焦**可靠性、安全、协作效率**，而非重做节点 UI 架构。

---

## 二、内部使用：优先优化（投入小、收益大）

### 1. 访问控制与 Key 集中管理 ⭐⭐⭐

**现状**：
- 服务默认绑定 `0.0.0.0:3000`，局域网内任意设备可访问
- API Key 明文存于 `API/.env`
- CORS 为 `allow_origins=["*"]`

**内部建议**（按团队规模选）：
- **5 人以内**：Nginx Basic Auth 或简单访问 Token 中间件
- **仅 IT 管理 Key**：普通成员只选模型，不在 UI 中暴露 Key 配置入口
- 生产环境优先 `127.0.0.1` + 内网反向代理，避免直接暴露到公网
- 日志中对 Key 脱敏

**收益**：防止 Key 泄露、误操作改配置、非授权访问。

---

### 2. 任务持久化（防重启丢任务）⭐⭐⭐

**现状**：`CANVAS_TASKS` 存于内存，服务重启后任务 404。

```python
CANVAS_TASKS: Dict[str, Dict[str, Any]] = {}
CANVAS_TASK_LOCK = Lock()
```

**建议**：
- **轻量方案（推荐）**：任务状态写入 `data/tasks/{task_id}.json`
- 完整方案（团队并发高时）：Redis + 后台 Worker

**收益**：长耗时生图/视频不再因重启、崩溃丢失；内部同事少踩坑。

---

### 3. 统一异步任务模式 ⭐⭐

**现状**：各节点 API 调用方式不一致：

| 节点 | 方式 | 无 GPU 场景 |
|------|------|-------------|
| API 生图 | 异步任务 + 轮询 ✅ | **主用** |
| 视频生成 | 同步长 HTTP ❌ | **主用**，优先改异步 |
| LLM | 同步长 HTTP ❌ | 常用，优先改异步 |
| ModelScope | 同步阻塞 ❌ | 可选（免费试玩） |
| ComfyUI | 异步任务 + 轮询 | **可忽略**（无本地 GPU） |
| RunningHub | 云端工作流 | 需要 Comfy 类能力时用 |

**问题**：视频/LLM 同步请求占满 Uvicorn worker，浏览器有超时风险；无 GPU 时团队几乎只走 API/视频节点，该问题更突出。

**建议**：视频、LLM、ModelScope 统一走「创建任务 → 轮询 / WebSocket 推送」，复用现有 `cascadeFetch` 与 `/api/canvas-image-tasks` 模式。

---

### 4. 画布内「从生成历史选择」⭐⭐

**现状**：历史记录在 `/api/history`，画布内复用需手动找文件或重新生成。

**建议**：在 Output 节点、图片节点、视频 Media 区增加「从历史选择」按钮，对接现有 history API。

**收益**：内部团队复用产出、减少重复调用 API 成本。  
**说明**：Liblib 有此入口；我们缺的是**便捷入口**，不是能力本身。

---

### 5. 内部工作流模板 ⭐⭐

**现状**：每次从零搭 Prompt → 生图/视频 → Output。

**建议**：
- 将常用链路存为画布模板 JSON，放在 `data/templates/`
- **无 GPU 推荐模板**：
  - `Prompt + Image → 视频生成 → Output`（如图生视频）
  - `Prompt → API 生成 → Output`（文生图/海报）
  - `Prompt → LLM → Prompt → API 生成`（文案再生成）
- 培训时说明：**不必配置 ComfyUI / LTX Director**，主用 API 生成、视频生成、LLM

**收益**：降低培训成本，统一团队出图规范。

---

### 6. 画布保存与渲染优化 ⭐

**现状**：
- 每次改动 500ms 防抖后**整份画布 JSON 全量 PUT**
- DOM 节点编辑器，多处 `innerHTML` 全量重绘（已有 `loading=lazy` 缩略图）

**建议**：
- 增量保存：只传变更的 nodes/connections（大画布时效果明显）
- 视口裁剪：100+ 节点时只渲染可见区域
- 单节点增量 DOM 更新，避免整画布 `renderAll()`

---

### 7. 备份与简单审计 ⭐

**建议**：
- 定时备份 `data/`、`assets/`、`API/.env`（勿提交 Git）
- 轻量审计：`data/audit.log` 记录时间、客户端 IP、模型、Provider（无需完整用户体系）

---

## 三、工程与稳定性（中期）

### 8. 后端模块化拆分

**现状**：路由、Provider、ComfyUI、画布 CRUD 均在 `main.py`。

**建议拆分**：

```
app/
  routes/       canvases.py, generate.py, providers.py, assets.py
  providers/    openai.py, apimart.py, modelscope.py, jimeng.py ...
  services/     canvas_service.py, task_service.py, comfy_service.py
  models/       pydantic schemas
  main.py       仅启动 + 挂载路由
```

**收益**：多人维护、加 Provider 时不互相影响。内部团队 2 人以上开发时值得做。

---

### 9. ComfyUI 相关（无 GPU 可跳过）

**现状**：`generate()` 同步 + `requests` + `QUEUE_LOCK`，经 `asyncio.to_thread` 包装。

**无 GPU / 无 ComfyUI 时**：无需投入此项；ComfyUI 设置、LTX Director 节点可对普通用户隐藏或文档注明「不使用」。

**若将来接入 RunningHub 云端工作流**：优先维护 RH 生成节点，仍不需要本地 GPU。

**有 GPU 后再考虑**：ComfyUI 改 `httpx.AsyncClient`、独立 worker 队列。

---

### 10. 文件并发与数据一致性

**现状**：多个 `Lock()` 保护 JSON，跨资源无事务。

**建议**：关键写操作加 `filelock`；素材库变大后可迁 SQLite（元数据）+ 文件系统（媒体）。

---

### 11. 错误处理与可观测性

**现状**：大量 `print()`；无 `/health` 端点。

**建议**：
- 统一 `logging` + 请求 ID
- 增加 `/health` 供内网监控
- 上游失败分类（超时 / 余额 / 内容审核），已有 `friendly_*_error_detail` 可继续完善

---

### 12. Provider 适配层策略化

**现状**：`generate_ai_image()` 内大量 `if protocol == ...` 分支。

**建议**：策略模式 + 注册表，新 Provider 只增文件不改主流程。

---

## 四、内部可选增强（按业务需要）

| 项 | 说明 | 何时需要 |
|----|------|----------|
| **@ 引用素材** | Prompt 内 `@` 联动画布资源 | 连线已够用；追求 Liblib 式输入体验时可做 |
| **Prompt 一键翻译** | 中英互译按钮 | 团队常用中文 Prompt、模型吃英文时 |
| **模型下拉加预估耗时** | 参考 Liblib 的 25s/50s 提示 | 提升等待预期，非刚需 |
| **画布版本快照** | 误改可回滚 | 多人同改一画布时 |
| **项目 > 画布层级** | 按客户/campaign 分组 | 画布数量多、需归档时 |
| **统一任务面板** | 进行中 / 失败 / 重试 | 并发生成多、需总览时 |

---

## 五、内部不必做

以下属于**商用 SaaS / 对标 Liblib 平台化**范畴，公司内部工具通常**不需要**：

- 积分 / 计费 / 订阅系统
- 公开分享链接、外网访问
- 社区 / 模板市场 / 创作者运营
- 完整多租户用户体系（10 人内共享 Token 即可）
- 对标 Liblib 的运镜预设库、导演台、角色一致性 ID（除非团队明确做短视频 IP）
- 本地 ComfyUI / GPU 部署与调优（**当前无 GPU，不需要**）
- React 全量重写前端（投入大，对内部小团队 ROI 低）
- 在未获授权前提下按 README 做商业封装（见项目著作权说明）

## 六、推荐内部架构（无 GPU）

```
┌─────────────────────────────────────────┐
│  普通内网服务器（无需显卡）               │
│  Infinite Canvas :3000                   │
│  画布 / 素材 / 历史存本地                │
└─────────────────────────────────────────┘
         ↑ 内网 + Nginx 密码 / Token
    ┌────┴────┬────────┬────────┐
  设计/运营   设计/运营   …      IT 管理员
                              （统一配置 Key）
         │ HTTPS 出网
         ▼
   APIMart / 火山 / ModelScope / RunningHub …
```

**配置原则**：
- 公司统一 API Key，由管理员维护 `API/.env`
- **不部署 ComfyUI**；复杂工作流用 RunningHub 云端
- 主用节点：**API 生成、视频生成、LLM、Output**
- `data/`、`assets/` 定期备份，勿暴露公网

详细步骤见 [`内部部署.md`](./内部部署.md)。

## 七、内部落地路线图（约 3–4 周）

| 阶段 | 内容 | 预估 |
|------|------|------|
| **第 1 周** | 按 [`内部部署.md`](./内部部署.md) 上线 + Key 集中管理 + 内网访问 | 必做 |
| **第 2 周** | 任务持久化 + 视频/LLM 改异步（无 GPU 时更关键） | 必做 |
| **第 3 周** | API/视频向工作流模板 + 备份脚本 + 团队培训（不讲 ComfyUI） | 建议 |
| **第 4 周** | 简单 audit.log + 统一 logging + 团队培训 | 建议 |
| **中期** | 拆分 `main.py`、视频/LLM 全异步、画布增量保存 | 开发人手充足时 |
| **长期** | SQLite 素材库、Provider 策略化、视口裁剪 | 按需 |

## 八、总结：内部使用最值得先做的 5 项

| 优先级 | 优化项 | 原因 |
|--------|--------|------|
| ⭐⭐⭐ | 访问控制 + Key 集中管理 | 内网默认暴露，内部安全第一 |
| ⭐⭐⭐ | 任务持久化 | 长任务不丢，减少运维投诉 |
| ⭐⭐ | 画布内从历史选择 | 提升复用、省 API 成本 |
| ⭐⭐ | 内部工作流模板 | 降培训成本、统一产出规范 |
| ⭐⭐ | 备份 + 简单审计 | 数据可恢复、问题可追溯 |

**不必优先**：重做节点内嵌 UI（已有）、积分系统、公开分享、前端框架重写。

---

## 九、与 Liblib.tv 的差异（内部视角）

| 维度 | Liblib.tv | 本项目（内部部署） |
|------|-----------|-------------------|
| 部署 | 云端 SaaS | 本地/内网 ✅ 数据不出公司 |
| 节点内嵌参数 | 有 | **有** ✅ |
| 多 API 可选 | 平台绑定 | **可换 Provider** ✅ |
| 本地 GPU | 不需要 | 需自建 Comfy（你们**不需要**） |
| 积分/成本展示 | 有 | 不需要（管理员看账单） |
| @ 引用 / 运镜预设 | 有 | 可选增强，非刚需 |
| 协作分享 | 云端账号 | WebSocket 多窗口 + 可选版本快照 |

**结论**：内部使用不必追 Liblib 的平台化能力；把**安全、可靠、模板、历史复用**做好，比照搬 Liblib 更合适。
